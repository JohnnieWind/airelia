# Airelia 开发守则

这份文件是本仓库的长期协作约定。后续每次实现需求时，优先遵守这里的结构、边界和验证规则，避免项目在功能增长后变得混乱。

## 开工前

- 每个需求开始前，先判断它应该落在哪一层：`packages/web`、`packages/server`、`electron`、`docs`，或多层协作。
- 动手前先给出简短的“落点判断”：涉及哪些模块、是否需要新建 feature/domain、是否需要 API 变更、是否需要测试。
- 复杂需求先写轻量设计文档，放在 `docs/superpowers/specs/` 下，说明目标、非目标、数据流、接口、错误处理和测试策略。
- 如果一个需求看起来会牵动多个独立子系统，先拆分成更小的可交付需求，不一次性把所有东西塞进代码里。

## 项目边界

- `electron/` 只负责桌面壳能力：窗口创建、preload、安全桥接、系统级能力。不要把业务逻辑写进 Electron 主进程。
- `packages/web/` 负责 React 前端体验、页面状态、组件、前端 API client 和前端测试。
- `packages/server/` 负责 Spring Boot 后端、业务领域、HTTP API、运行时集成和后端测试。
- `docs/` 放设计、计划、架构说明和较长期的工程决策。
- 跨层通信必须通过明确接口完成，例如 HTTP API、preload 暴露的极小 API，不要让前端依赖后端内部实现细节。

## 代码组织

- 新功能优先按功能切片组织，而不是把所有代码堆进通用文件。
- 前端功能增长后，优先使用类似 `packages/web/src/features/<功能名>/` 的结构，内部再放组件、状态、API、测试。
- 后端功能增长后，优先使用类似 `packages/server/src/main/java/com/airelia/server/<领域名>/` 的结构，按领域放 controller、service、model、dto。
- 文件职责要单一。一个文件如果开始同时承担 UI、请求、状态编排和数据转换，应该拆分。
- 控制器保持薄：接收请求、校验输入、调用服务、返回响应。业务规则不要长期堆在 controller 里。
- 共享抽象不要过早创建。重复出现到第 2-3 次，再考虑抽到 `shared`、`common`、`utils` 或服务层。
- 不建立“万能 utils”或“杂物 common”。每个共享模块都要有清楚边界和命名。

## 变更纪律

- 每次改动尽量围绕一个明确目标，不顺手做无关重构。
- 遇到脏工作区时，不要回滚用户或其他工具产生的改动；只处理当前任务需要触碰的文件。
- 不把临时代码、调试输出、无用 TODO 留在主路径里。
- 配置文件要保持可运行。JSON 文件不能添加注释，避免破坏格式。
- 注释应该解释意图、边界、关键决策和非显然逻辑，不要逐行翻译语法。

## API 与数据流

- 新增接口前先明确请求体、响应体、错误状态和前端消费方式。
- 前端请求后端要经过集中 API client 或 feature 内的 API 模块，不要在组件深处散落裸 `fetch`。
- 后端响应结构要稳定，字段名变化需要同步更新前端类型和测试。
- 错误处理要作为需求的一部分考虑：加载中、失败、空状态、校验失败都要有明确行为。

## 前端规范

- 保持界面第一屏就是可用产品体验，不做无意义的营销式落地页。
- 组件优先服务工作流：状态清晰、操作路径短、反馈明确。
- UI 文案、按钮、输入和卡片布局要适合桌面应用，避免装饰性过强或信息密度过低。
- 新增交互时，同时考虑 loading、error、empty、disabled 和成功状态。
- 使用项目已有的 Tailwind、React、lucide-react 习惯，不随意引入新的 UI 框架。

## 后端规范

- 使用 Spring Boot 现有风格组织接口和测试。
- 请求体校验使用 Jakarta Validation，例如 `@Valid`、`@NotBlank`。
- 外部运行时或 Agent 集成要包在清晰服务边界内，不让控制器直接承载复杂集成逻辑。
- 需要可配置的值放进 `application.yml` 或清晰的配置类，不把环境差异硬编码在业务代码里。

## 测试与验证

- 前端 UI 行为使用 Vitest 和 React Testing Library 覆盖关键路径。
- 后端 API 使用 Spring Boot Test 和 MockMvc 覆盖请求、响应和校验行为。
- 每个需求至少运行与改动相关的测试；跨前后端改动优先运行 `npm test`。
- 声称完成前，检查 `git diff --check`，并确认没有格式、空白或明显临时代码问题。
- 如果测试无法运行，要说明原因和剩余风险。

## 收尾检查

完成一个需求前，快速自查：

- 这个需求的代码是否落在正确层级？
- 是否引入了不必要的共享抽象？
- 是否有文件变得过大或职责混杂？
- API 类型、前端调用和测试是否同步？
- 错误状态、空状态和边界输入是否被考虑？
- 是否留下临时代码、无用注释或调试输出？

目标不是让项目一开始就复杂，而是让它每次增长时都有方向感。
